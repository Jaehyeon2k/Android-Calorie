# 테스트 계획 (Test Plan)

## 1. 테스트 전략
* **단위 테스트 (Unit Test):** 복잡한 비즈니스 로직(특히 칼로리 비례 계산 로직, BMR 계산 로직)이 정확한지 검증한다.
* **UI 테스트 (Instrumented Test):** Compose 화면 전환, 입력 폼 검증, 키보드 동작 등을 디바이스에서 검증한다.
* **수동 통합 테스트 (Manual E2E Test):** 음식 검색 -> 등록 -> 다이어리 기록으로 이어지는 전체 흐름을 테스트한다.

## 2. 주요 테스트 케이스 (Test Cases)

| TC-ID | 기능 | 테스트 시나리오 | 예상 결과 |
|-------|------|----------------|-----------|
| TC-01 | 칼로리 계산 | 100g 당 120kcal인 음식을 50.5g 입력 | 60kcal로 비례 계산되어 저장됨 |
| TC-02 | 직접 만든 음식 | 빈 재료가 포함된 상태로 저장 시도 | 빈 재료는 무시되고 유효한 재료만 합산됨 |
| TC-03 | API 검색 필터링 | '닭가슴살' 검색 시 | '샌드위치_닭가슴살'(240kcal)이 아닌 순수 '닭가슴살'(약 110kcal)이 매칭됨 |
| TC-04 | API 검색 실패 | '아무음식' 등 미존재 음식 검색 | API 결과가 없음을 감지하고 칼로리 입력 필드를 0으로 초기화 |
| TC-05 | 키보드 숨김 | 입력 필드 밖의 빈 공간을 터치 | 포커스가 해제되며 키보드가 내려감 |
