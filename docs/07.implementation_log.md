# 구현 이력 (Implementation Log) — 스마트 칼로리 트래커

## 1. 주요 기술 스택
- **Language:** Kotlin
- **UI Toolkit:** Jetpack Compose (Material 3)
- **Local Database:** Room Database (Offline First)
- **Asynchronous & State:** Kotlin Coroutines & StateFlow
- **Network:** HttpURLConnection / JSONObject (외부 라이브러리 최소화)
- **Image Loading:** Coil Compose

## 2. 주요 구현 내역 및 트레이드오프

### 2.1 [Phase 1] 아키텍처 및 뼈대 구축
- **결정:** MVVM 패턴(Model-View-ViewModel) 도입. UI는 상태를 갖지 않고 `StateFlow`를 Observe하며, ViewModel이 비즈니스 로직을 처리.
- **트레이드오프:** 초기에는 ViewModel에서 Room DAO를 직접 호출하는 방식으로 빠르게 프로토타이핑하였으나, 기능(API, Room)이 비대해지면서 최종적으로 `FoodRepository` 계층을 도입하여 데이터 소스를 추상화함.

### 2.2 [Phase 2] 오프라인 퍼스트 DB 설계
- **결정:** 음식 정보(`FoodItemEntity`), 재료(`IngredientEntity`), 식사 기록(`MealRecordEntity`)을 철저히 분리. 
- **이유:** 사용자가 "직접 만든 요리"를 등록할 때 재료 구성을 별도로 저장/수정할 수 있도록 정규화(Normalization) 하였으며, 삭제 시 `CASCADE` 옵션을 주어 하위 재료도 자동으로 지워지도록 설계.
- **트레이드오프:** 요리 조회 시 매번 재료 테이블과 `JOIN`을 수행해야 하는 성능 비용이 발생하지만, 개인용 로컬 DB(Room) 수준에서는 오버헤드가 극히 미미하며 데이터 무결성을 유지하는 것이 더 이득이라 판단함.

### 2.3 [Phase 3] API 연동 및 데이터 필터링
- **결정:** 식약처 공공데이터포털(data.go.kr)의 OpenAPI를 Retrofit 등의 라이브러리 없이 `HttpURLConnection`으로 직접 파싱하여 경량화 구현.
- **문제와 해결:** 공공데이터 특성상 "닭가슴살" 검색 시 가공식품("닭가슴살 샌드위치 300kcal")이 먼저 나오는 문제가 발생함. 검색 결과를 20개 가져와서 문자열 길이를 측정하고, "튀김, 가공, 샌드위치" 키워드에 페널티(+100점)를 부여하여 가장 순수한 원물 식재료를 매칭하는 지능형 스코어링(Scoring) 알고리즘을 도입함.

### 2.4 [Phase 4] 요리백과 기능 및 로컬 캐싱
- **결정:** 외부 레시피 API 데이터를 검색만 하고 끝나는 것이 아니라, `RecipeEntity` 테이블을 만들어 로컬에 저장(Import)하도록 구현.
- **이유:** 앱의 핵심 가치인 "오프라인 기능 보장(Offline-First)"을 위해, 한 번 가져온 레시피는 언제든 비행기 모드에서도 꺼내어 식단에 등록할 수 있도록 영속화함.

### 2.5 [Phase 5] UX 폴리싱 및 입력 정밀도 강화
- **사진 첨부:** 사용자가 음식 사진을 등록할 때 권한 문제가 복잡해지는 것을 막기 위해 안드로이드 최신 `ActivityResultContracts.PickVisualMedia` (Photo Picker)를 사용하여 안전하게 갤러리 접근.
- **소수점 계산:** 무게 입력 시 정수형이 아닌 `toDoubleOrNull()`을 사용하여 0.5g 입력 시에도 오차 없이 정확한 소수점 칼로리가 도출되도록 수식 리팩토링.
